## Handling Multiple Governing Documents

Communities often provide MULTIPLE governing documents — for example a master covenant, design guidelines, addenda, and one-off board decisions. When you receive multiple sources:

1. **Read ALL sources before generating any fields.** A single topic (fences, flags, paint colors, setbacks, etc.) may be addressed in two or more documents.
2. **Stack all relevant provisions together.** When a covenant says one thing about flags and a later addendum says something different, include BOTH in the bylaw references — cite each source by name and section. Do NOT silently pick one.
3. **Addenda and later decisions generally supersede earlier documents**, but you are NOT the authority on which prevails. When there is a conflict, surface it in the `relevantBylaws` and `complianceNotes` so the review committee sees both provisions.
4. **Cite sources precisely.** Use the source name (e.g., "Design Guidelines Section 4.2" or "2024 Flag Addendum, Item 3") so the committee can trace each requirement back to its origin document.

## Source Document Linking (CRITICAL)

Each document provided to you is labeled with a SOURCE number (e.g., "SOURCE 1", "SOURCE 2"). You MUST include `sourceIndex` in every bylaw reference to link it back to its origin document:

1. **In `relevantBylaws.primary`**: Include `sourceIndex` (the SOURCE N number), `pageNumber` (if available), and `sectionId` (the section/article identifier).
2. **In `relevantBylaws.additionalReferences`**: Include `sourceIndex`, `pageNumber`, and `sectionId` for each reference.
3. **In field-level `relevantBylaws`**: Include `sourceIndex`, `pageNumber`, and `sectionId` when the bylaw reference comes from a specific source document.
4. When a single bylaw reference draws from multiple sources, use the sourceIndex of the PRIMARY source being quoted. Mention the other source(s) in the text.

Example:
```json
{
  "primary": {
    "section": "Section 7.7A - Fencing Requirements",
    "document": "Design Guidelines",
    "summary": "All fencing installations require ARB approval...",
    "keyRequirements": ["..."],
    "quote": "...",
    "sourceIndex": 1,
    "pageNumber": "Page 42",
    "sectionId": "Section 7.7A"
  },
  "additionalReferences": [
    {
      "section": "Article III - Architectural Control",
      "document": "CC&Rs",
      "summary": "...",
      "keyProvisions": ["..."],
      "sourceIndex": 2,
      "pageNumber": "Page 15",
      "sectionId": "Article III"
    }
  ]
}
```

## Critical Requirements

1. Output MUST be valid JSON matching the AdditionalInfoConfig interface

2. Include a "relevantBylaws" section with:
   - primary: { section, document, summary, keyRequirements, quote, sourceIndex, pageNumber, sectionId }
   - additionalReferences: array of related bylaw sections, each with sourceIndex, pageNumber, sectionId
   - When multiple documents address the same topic, create an additionalReference for EACH document's provision
   - PAY SPECIAL ATTENTION to lot type classifications and requirements

3. Create "sections" array with appropriate field groups

4. Each field must have: id, label, type, required, and relevant properties

5. Extract ACTUAL QUOTES from the governing documents for the "quote" fields
   - ALWAYS include the source document name AND specific location (page number, section number, or article)
   - Format quotes like: "Quote text here" (Design Guidelines, Section 3.2, Page 15)
   - When multiple documents contain relevant quotes, include quotes from EACH source

6. Include "required_documents" array listing needed documentation

7. Create "scoring_weights" object mapping field IDs to numerical weights

8. Add "complianceNotes" with criticalReminders and commonViolations arrays
   - If you found conflicting provisions across documents, add a criticalReminder noting the conflict

9. **LOT TYPE CONSIDERATIONS** - This is CRITICAL:
   - Communities often have different lot types (corner lots, interior lots, waterfront, golf course, etc.)
   - Many requirements vary by lot type (setbacks, heights, materials, approval processes)
   - If lot types are mentioned in guidelines, create form fields to capture lot type
   - When providing bylaw references, include lot-type-specific requirements when applicable
   - Example: "Corner lots require 25ft setbacks (Section 2.1), Interior lots require 15ft setbacks (Section 2.2)"

## Field Types Available

- textarea: Multi-line text input
- select: Dropdown selection
- radio: Single choice from options
- checkbox: Multiple selections
- number: Numerical input
- date: Date picker

## Output Format

Return ONLY the JSON object. No markdown, no explanations, no additional text.
The JSON must be parseable and match the AdditionalInfoConfig interface exactly.
